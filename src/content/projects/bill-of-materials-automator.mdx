## Context & Background

Solar designs are built from a finite set of hardware: panels, batteries, inverters, EV chargers. But the bill of materials (BOM) also needs a long tail of extra hardware and service items that depend on the site (roof type, slope, travel distance) and the design itself (number of panels, inverter type, and so on).

Encoding all of that in application code would have meant a constant maintenance burden for engineers and very little flexibility for the people who actually own the pricing and BOM logic. The Bill of Materials Automator is a rules engine that lets those users define rules which automatically add the right items to a BOM based on the design and site.

## Implementation

I led the work to grow this from a single rule type into a small rules engine that non-developers could safely author against. When I joined, it could already add components based on the presence of other components, modelled as a directed acyclic graph. I designed and delivered three further rule types (conditional, fixed, and bundle rules), along with the guardrails that make operator-authored rules safe to run.

At a high level, the engine loads the active rules for a project, groups them by type, and runs them in a defined order. Each type decides *what* gets added and *when*.

**Conditional rules** are powered by a small, dependency-free expression language. Operators can express conditions and quantities as formulas (mapping over the rows of a design, rounding up to whole units, and so on) rather than waiting on an engineer to ship a change.

**Fixed rules** add hardware and fees, and they surfaced a subtle ordering problem. Rules that add hardware need to run early, so later rules can build on what they add. But some rules calculate their quantities from running totals (like total hardware or installation cost) that only exist once most of the BOM has been built. I split fixed rules into two passes based on whether a rule depends on those totals, so each rule runs at the only point where its inputs are actually available.

### Bundles

Bundles represent a set of components that are sold and priced together (a kit that is really a dozen smaller parts) under a single parent item. The interesting part is that bundle handling runs *in reverse*: instead of adding items to a design, it recognises when a finished design already contains the components that make up a known bundle and collapses them into a single priced line.
